# Requirements

## 目的

Fes Route は、音楽フェスやサーキットイベントの参加者が、1つの canonical なイベント / タイムテーブルデータをもとに、自分の回り順を作成し、友人と共有・比較できる web app。

特に次の3つを簡単にする。

- 公式または信頼できるタイムテーブルを見つける
- 見たいアーティストを選んで自分の予定を組む
- 友人と予定を共有し、合流や別行動を判断する

## 基本方針

- 1つの実イベントに対して、アプリ上のイベントデータは1つに寄せる。
- 閲覧・個人予定作成・共有にログインを必須にしない。
- canonical データの品質は管理サイドの承認フローで守る。
- 初期のイベント / タイムテーブル入力は管理サイドの手動運用を前提にする。
- 個人ユーザーもイベント登録・修正提案はできる。
- 各種チケットサイトからのイベント情報取得は、まず管理サイドの入力補助として扱う。
- タイムテーブルは管理サイドが X、公式サイト、公式画像、PDF などから取得して設定する。
- 位置情報共有は扱わない。
- スマホ web を主導線にする。

## ユーザー種別

### フェス参加者

- イベント情報とタイムテーブルを見る
- 見たいアーティストを自分の予定に追加する
- 予定の被りを確認する
- 自分の予定を共有する
- グループ内で友人の予定と比較する
- イベント追加や修正を提案する

### グループ参加者

- 招待リンクからグループに参加する
- ログインせず表示名だけで参加できる
- 同じアーティストを見る予定の人を確認する
- 合流できそうな空き時間を確認する
- 別行動になりそうな時間帯を確認する

### 管理者

- canonical event を作成・編集する
- チケットサイトからイベント基本情報を取得・転記する
- X、公式サイト、公式画像、PDF などからタイムテーブルを登録する
- ユーザーからのイベント追加・修正提案を確認する
- 提案を承認 / 却下する
- 重複イベントを統合する
- イベントを公開 / 非公開 / アーカイブする

## 機能要件

### イベント一覧 / 詳細

- 公開中のイベント一覧を表示できる。
- イベント詳細ページを開ける。
- 各イベントは安定した公開 URL を持つ。
- 各実イベントは1つの canonical event として管理する。
- イベントには下書き、公開、要確認、アーカイブの状態を持たせる。
- イベントには情報取得元 URL と取得日時を保存する。

必須項目:

- イベント名
- 開催日
- 終了日、複数日開催の場合
- 会場名
- 都道府県または地域
- 公式サイト URL、ある場合
- チケットサイト URL、ある場合
- 説明文、任意
- メイン画像、任意
- 情報取得元
- 公開状態

### 重複防止

1イベント 1データを守るため、イベント作成前に重複候補を検出する。

重複判定に使う情報:

- イベント名の類似
- 開催日の一致または重なり
- 会場の一致
- 公式 URL の一致
- チケットサイト URL の一致
- 主催者 URL の一致、取得できる場合

必要な挙動:

- ユーザーが既存イベントに近いイベントを投稿した場合、既存イベントを先に提示する。
- 管理者は重複イベントを統合できる。
- 統合時は source URL、修正提案、履歴をできるだけ保持する。
- 統合後も canonical event の公開 URL は可能な限り維持する。

### イベント入力

初期入力は管理サイドが手動で行う。

個人ユーザーは次を投稿できる。

- 新規イベント登録提案
- イベント情報の修正提案
- タイムテーブルの修正提案

ユーザー投稿は canonical data を直接上書きしない。必ず reviewable proposal として保存する。

提案に持たせる情報:

- 対象イベント、ある場合
- 提案種別
- 変更対象フィールド
- 変更前後の差分
- 情報源 URL、任意だが推奨
- 投稿者の表示名、任意
- 作成日時
- レビュー状態
- 管理者メモ、任意

### チケットサイト連携

各種チケットサイトからイベント情報を取得する。MVP では完全自動収集ではなく、管理者が URL を入力して取得できる項目を補助入力する形でよい。

対象候補:

- e+
- チケットぴあ
- ローチケ
- LivePocket
- Zaiko
- その他イベント掲載サイト

最低限取得したい項目:

- イベント名
- 開催日
- 会場
- 開場時刻、ある場合
- 開演時刻、ある場合
- チケット URL

取得できるとよい項目:

- 出演者
- 主催者
- チケット価格
- 販売期間
- イベント説明
- サムネイル画像

必要な挙動:

- 管理者がチケットサイト URL を入力できる。
- アプリは取得できる項目を自動補完する。
- 管理者は保存前に取得内容を確認・編集できる。
- 取得に失敗した場合は手入力に戻せる。
- 取得元 URL と取得日時を保存する。
- 対応サイトごとに import adapter を分ける。

### タイムテーブル管理

タイムテーブルは管理サイドが作成・保守する。

情報源:

- 公式サイト
- 公式 X アカウント
- 公式タイムテーブル画像
- 公式 PDF
- チケットサイト掲載情報
- 管理者が確認した情報

必須項目:

- アーティスト名
- ステージ
- 開始時刻
- 終了時刻
- 日付
- 情報源、任意だが推奨
- 状態

状態:

- scheduled
- changed
- cancelled
- tentative
- unannounced

必要な挙動:

- 管理者はタイムテーブル枠を作成・編集・削除できる。
- 管理者は source URL または source note を紐づけられる。
- タイムテーブル変更履歴を残す。
- ユーザーは修正提案を送れる。
- 修正提案は管理者承認後に canonical data へ反映する。
- 変更・中止された枠はユーザーに分かるように表示する。

### 個人予定

ログインなしで予定を作成できる。

必要な挙動:

- タイムテーブル枠を「見たい」に追加できる。
- タイムテーブル枠を「迷い中」に追加できる。
- 自分の予定から削除できる。
- 個人メモを残せる。
- 選択した予定の時間被りを検出する。
- イベント当日は「今やっている予定」「次の予定」を表示する。
- 共有 URL を発行できる。

保存方針:

- ログインなしの個人予定は local storage を基本にする。
- 共有する場合は share id を発行する。
- 将来ログインを追加できるよう、内部的には anonymous user id を持てる設計にする。

### グループ共有

ログインなしでグループを作成できる。

必要な挙動:

- グループ招待リンクを作成できる。
- メンバーは表示名だけで参加できる。
- メンバーは自分の予定をグループに共有できる。
- アーティストごとに誰が見る予定か表示する。
- 同じアーティストを見るメンバーを表示する。
- 予定が分かれる時間帯を表示する。
- 合流できそうな空き時間を表示する。

MVP では不要:

- チャット
- リアルタイム位置共有
- フレンド機能
- SNS 的なタイムライン

### タイムテーブル表示

必要な表示:

- 時間軸表示
- ステージ別表示
- アーティスト一覧
- 自分の予定
- グループ比較
- 当日の now / next 表示

検索 / 絞り込み:

- アーティスト名検索
- ステージ絞り込み
- 日付絞り込み
- 見たい / 迷い中 / 未選択で絞り込み
- グループメンバーが見たいアーティストで絞り込み

### 認証 / ID

ログインは必須にしない。

必要な挙動:

- 公開イベントはログインなしで閲覧できる。
- 個人予定はログインなしで作成できる。
- グループ共有は招待リンクで成立する。
- 表示名はグループ単位で任意入力にする。
- 不要な個人情報は収集しない。
- 将来ログインを追加しても anonymous data を移行できる設計にする。

### 管理画面

canonical data を維持するため、管理画面が必要。

必要な操作:

- イベント作成
- イベント編集
- イベント公開
- イベント非公開
- イベントアーカイブ
- イベント提案レビュー
- 修正提案レビュー
- 提案承認
- 提案却下
- 重複イベント統合
- イベント source 管理
- タイムテーブル作成
- タイムテーブル編集
- タイムテーブル変更履歴確認

## 非機能要件

### 対応環境

- スマホ web 優先
- PWA 対応を見据える
- 管理画面は desktop でも使いやすくする
- 最新の Safari / Chrome で動作する

### パフォーマンス

- モバイル回線でもイベント詳細を素早く表示する。
- 一般的なフェス規模のタイムテーブル操作は即時に感じられる速度にする。
- 大規模イベントでもステージ / 時間帯の絞り込みで破綻しない。

### オフライン / 低通信環境

- 最後に読み込んだイベントデータは可能な限りオフラインでも閲覧できる。
- 個人予定はオフラインでも確認できる。
- データが古い可能性がある場合は UI で示す。

### プライバシー

- コア利用にアカウント登録を要求しない。
- 位置情報を収集しない。
- 共有 URL やグループ招待リンクを持つ人だけが該当データを見られる。
- 個人予定はユーザーが共有するまで公開しない。
- グループ招待リンクは bearer access として扱う。

### データ品質

- canonical data の変更は source または admin action に紐づける。
- ユーザー編集は直接反映せず proposal として扱う。
- 統合履歴を残す。
- タイムテーブル変更履歴を残す。
- ユーザーから見て「なぜ変更されたか」を追える最低限の source を残す。

## MVP スコープ

### In

- 管理者によるイベント作成
- 管理者によるタイムテーブル作成
- チケットサイト URL からのイベント情報補助取得
- 公開イベント詳細
- タイムテーブル閲覧
- ログインなしの個人予定作成
- 共有 URL による予定共有
- 招待リンクによるグループ作成
- グループ内予定比較
- ユーザーによるイベント登録提案
- ユーザーによる修正提案
- 管理者による提案レビュー
- 重複イベント候補検出

### Out

- ログイン必須化
- GPS / ライブ位置共有
- チャット
- 決済
- ネイティブアプリ
- 全チケットサイトの完全自動巡回
- SNS 的なフレンド機能
- 高度なレコメンド
- プレイリスト生成
